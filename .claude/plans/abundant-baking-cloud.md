# V3.3 Final Fix Plan — 12-Agent Validated

## Context

This plan was audited by 12 specialized agents (10 validation + 2 post-training) with Perplexity MCP verification. Every LightGBM parameter, Optuna range, RAM calculation, and philosophy decision has been validated against external sources.

**Root causes of 4 failed training runs identified:**
1. `OMP_NUM_THREADS=4` in launch command, never removed by cloud_run_tf.py → LightGBM capped at 4 threads
2. `feature_pre_filter` not set → LightGBM defaults True, silently kills rare esoteric features
3. No 16-DB verification → missing databases cause silent feature degradation
4. Stale v3.2 artifacts (features_*_all.json, cross_names) reused without rebuild → 173 new features missing

**Perplexity-validated surprise:** `is_enable_sparse=True` + dense data does NOT cause single-threaded training. LightGBM auto-detects dense input. The real culprit was always `OMP_NUM_THREADS=4`.

---

## MATRIX THESIS (include in every fresh Claude context)

This is NOT a conventional trading bot. The edge is 2.9 MILLION sparse binary cross features generated by cross-multiplying esoteric signals (gematria, astrology, numerology, space weather, tweet decodes) with technical analysis indicators. LightGBM's EFB (Exclusive Feature Bundling) bundles mutually exclusive binary features — architecturally why LightGBM is used. NaN = "missing signal" (LightGBM learns optimal split direction). 0 = "value is zero" (DIFFERENT meaning). NO feature can EVER be filtered or removed. The model decides via tree splits.

## 15m: Row-Partitioned Boosting (UNBLOCKED — Perplexity-validated design)

### Why 15m needs special handling

Agent 4 confirmed using ACTUAL NNZ data from cloud NPZ files:
- 1w: 40M NNZ (1.9% of int32 limit)
- 1d: 481M NNZ (22.4%)
- 1h: 1.89B NNZ **(87.9% — dangerously close)**
- **15m estimated: 16-40 BILLION NNZ (7.6-19x OVER int32 limit)**

LightGBM's C++ core uses int32 indices. Exceeding 2^31 NNZ produces **silent garbage predictions** (GitHub #1689). No compile flag exists for int64.

### Why EFB makes training FAST once data is loaded

Agent 2 discovered: EFB reduces 10M binary features to **~39,000 bundles** (hardcoded `max_bin_per_group=256`). After EFB, LightGBM trains on ~39K features — **training itself takes minutes, not hours**. The bottleneck is Dataset construction, not training.

Sparse IS multi-threaded (our previous belief was wrong). `force_col_wise=True` parallelizes histogram building across features via OpenMP.

### Implementation: Row-Partitioned Incremental Boosting

```python
# For each CPCV fold:
def train_fold_partitioned(X_train_csr, y_train, params, total_rounds):
    """Train with row partitioning when NNZ > 1.8B (safety margin below 2^31)."""

    # 1. Calculate NNZ-balanced chunk boundaries
    row_nnz = np.diff(X_train_csr.indptr)
    cumulative = np.cumsum(row_nnz)
    NNZ_LIMIT = 1_800_000_000  # 84% of int32 max

    chunk_boundaries = [0]
    for i, total in enumerate(cumulative):
        if total - cumulative[chunk_boundaries[-1]] > NNZ_LIMIT:
            chunk_boundaries.append(i)
    chunk_boundaries.append(len(row_nnz))
    n_chunks = len(chunk_boundaries) - 1

    # 2. SHUFFLE rows (critical for approximation quality)
    shuffle_idx = np.random.permutation(X_train_csr.shape[0])
    X_train_csr = X_train_csr[shuffle_idx]
    y_train = y_train[shuffle_idx]

    # 3. Build reference dataset from chunk 0 (defines bin edges)
    chunk0 = X_train_csr[chunk_boundaries[0]:chunk_boundaries[1]]
    y_chunk0 = y_train[chunk_boundaries[0]:chunk_boundaries[1]]
    ref_dataset = lgb.Dataset(chunk0, label=y_chunk0, free_raw_data=False)
    ref_dataset.construct()

    # 4. Train chunk 0
    rounds_per_chunk = total_rounds // n_chunks
    booster = lgb.train(params, ref_dataset,
                        num_boost_round=rounds_per_chunk,
                        keep_training_booster=True)

    # 5. Continue with remaining chunks
    for i in range(1, n_chunks):
        chunk_i = X_train_csr[chunk_boundaries[i]:chunk_boundaries[i+1]]
        y_chunk_i = y_train[chunk_boundaries[i]:chunk_boundaries[i+1]]
        ds_i = lgb.Dataset(chunk_i, label=y_chunk_i, reference=ref_dataset)
        booster = lgb.train(params, ds_i,
                            num_boost_round=rounds_per_chunk,
                            init_model=booster,
                            keep_training_booster=True)

    return booster
```

**Key settings (Perplexity-validated):**
- `learning_rate`: CONSTANT across all chunks (do NOT reduce)
- `num_boost_round`: SPLIT across chunks (`total // n_chunks` per chunk)
- `min_data_in_leaf`: sized for FULL dataset, not rescaled per chunk
- `bagging_fraction`: 1.0 (disable — already implicit bagging by partition)
- `reference=ref_dataset`: ensures consistent bin boundaries across chunks
- `keep_training_booster=True`: avoids serialize/deserialize overhead

**Limitations (documented):**
- NOT equivalent to full-data training — approximation only
- Early trees never see later chunk rows for split decisions
- Order-dependent (mitigated by shuffling)
- Each chunk's trees are locally optimal for that partition
